Control Unit State Diagram – Design Review Issues

1. Load 동작 지속 문제 (s2)
- 현재 구조에서 s2 상태 동안 load_reg = 1이 계속 유지됨
- 즉, 단일 클럭 로드가 아니라 다중 클럭 동안 반복 로드 발생
- 영향:
  입력 A가 변할 경우 의도치 않은 재로드 가능
  FSM에서 “로드 이벤트”와 “대기 상태”가 분리되지 않음
- 결론:
  과제 수준에서는 허용 가능
  실무에서는 1-cycle pulse 형태 권장

2. Load_A 조건 미사용 (s2 → s3)
- s2에서는 Load_A를 보지 않고 S만으로 다음 상태 전이
- 즉, Load_A 이후 상태 유지 조건이 명확히 정의되지 않음
- 영향:
  Load_A와 S의 타이밍 관계가 명확하지 않으면 비직관적 동작 발생 가능
- 결론:
  단순 FSM에서는 허용 가능
  입력 프로토콜을 명확히 정의하는 것이 중요

3. Done 출력 지속 시간 (s4)
- s4에서 Done = 1, 다음 클럭에 바로 s1 복귀
- Done이 1-cycle pulse로 동작
- 영향:
  외부에서 Done을 레벨 신호로 기대하면 인식 실패 가능
- 결론:
  pulse형 Done으로 사용할 경우 문제 없음
  필요 시 Done 유지 state 추가 고려 가능

4. Reset 상태 표현 방식
- reset → s1(reset) 형태로 표현됨
- 일반 FSM에서는 reset을 별도 state로 두지 않고 initial state로 설정
- 영향:
  기능적 문제 없음
  다이어그램 표현 방식의 표준성 문제
- 결론:
  s1을 initial state (IDLE)로 정의하는 것이 일반적

5. Start 신호 S의 레벨 유지 문제
- S를 레벨 신호로 사용 (S=1 동안 계속 유효)
- 영향:
  S가 여러 클럭 동안 1이면,
  FSM이 완료 후 자동으로 다음 동작을 재시작할 수 있음
- 예:
  Done 이후 s1 복귀 → s2 진입 → S=1 유지 → 즉시 s3 재진입
- 결론:
  S는 1-cycle start pulse로 사용하는 것을 전제로 설계됨
  필요 시 S=0을 기다리는 state 추가 가능

## 모든 state에서 모든 output 정의해야 할까?

그렇게 생각하기 쉬운데, **“이전값이 같을 거라고 보장할 수 없다는 게 문제”**입니다.

핵심만 정리합니다.

---

지금 구조에서 생기는 실제 문제:

예를 들어 흐름이 이렇게 간다고 가정하면

* s11: Done = 1
* 다음 클럭 → s10 진입

그런데 s10에서 Done을 정의 안 했으니까

→ Done은 그대로 1 유지됩니다

즉,
**counting 중인데 Done=1 상태가 유지되는 이상한 상황 발생**

---

같은 논리로

* s10에서 shift_en = 1
* 다음에 s01로 갔는데 shift_en 안 꺼주면

→ shift 계속 돌아감

---

즉 결론은 이겁니다:

이전값이 같으면 괜찮은 게 아니라
**이전 state가 무엇이었는지에 따라 결과가 달라지는 구조 자체가 위험**

---

왜 이게 중요한가

FSM은 원칙적으로

* “현재 state만으로 출력이 결정”되어야 함 (Moore)

근데 지금 구조는

* 현재 state + 이전 state에 의존

→ Moore 머신이 아니라 **implicit latch behavior**가 됨

---

정리

* 이전값 유지 자체는 합법적인 동작임
* 하지만 설계 관점에서는
  → “state만으로 출력이 결정되지 않는다”는 게 문제

---

그래서 실무 기준은 이렇게 딱 정리됩니다

* **모든 state에서 모든 output을 정의한다**
* 또는
* **기본값을 먼저 깔고 필요한 것만 override한다**

---

한 줄 요약

“이전값이 같으면 괜찮다”가 아니라
**“이전값에 의존하게 만들면 안 된다”**가 맞습니다.
