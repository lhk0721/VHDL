# Lab 01 작업 안내서

## 1. 이 과제에서 해야 하는 일

이번 과제의 목표는 `two-phase handshake transmitter`를 만드는 것입니다.

쉽게 말하면:

- 보내는 쪽이 데이터를 준비한다.
- `req` 신호를 바꿔서 "지금 보낼게요"라고 알린다.
- 받는 쪽이 `ack` 신호를 같은 값으로 바꿔서 "받을 준비가 됐어요"라고 답한다.
- 보내는 쪽은 `ack`가 맞춰질 때까지 기다린다.

이 과제에서는 이 동작을 `상태 기계(FSM)` 방식으로 구현해야 합니다.

---

## 2. 먼저 이해해야 할 핵심 개념

### Handshake란?

두 회로가 서로 신호를 주고받으면서 안전하게 데이터를 전달하는 방법입니다.

### Two-phase handshake란?

`req`를 0에서 1로, 또는 1에서 0으로 "토글(toggle)" 하면서 전송 요청을 표시하는 방식입니다.

중요한 점:

- `req`는 항상 같은 방향으로만 바뀌는 것이 아닙니다.
- 이전 값이 0이면 1로, 1이면 0으로 뒤집습니다.
- 받는 쪽은 `ack`를 `req`와 같은 값으로 맞춰 응답합니다.

### FSM이란?

FSM은 회로의 동작을 "상태"로 나누어 순서대로 처리하는 방식입니다.

예를 들면:

- 대기 상태
- 데이터 저장 상태
- 요청 신호 변경 상태
- 응답 대기 상태

이렇게 나누면 회로 흐름을 보기 쉽고, 실수도 줄일 수 있습니다.

---

## 3. 입력과 출력

### 입력 신호

- `Data_to_send`: 보내려는 8비트 데이터
- `ready`: 데이터가 준비되었는지 알려주는 신호
- `ack`: 받는 쪽의 응답 신호
- `clk`: 클록
- `reset_n`: 리셋 신호

### 출력 신호

- `Data`: 실제로 내보내는 8비트 데이터
- `req`: 전송 요청 신호

---

## 4. 전체 동작을 아주 쉽게 보면

1. `ready = 1`이 될 때까지 기다립니다.
2. 보내려는 데이터를 내부 레지스터에 저장합니다.
3. `req` 값을 반대로 바꿉니다.
4. 받는 쪽이 `ack`를 같은 값으로 바꿀 때까지 기다립니다.
5. 확인이 끝나면 다시 처음 상태로 돌아갑니다.

짧게 정리하면:

`준비 확인 -> 데이터 저장 -> req 변경 -> ack 기다림 -> 다시 대기`

---

## 5. 상태를 어떻게 나누면 좋은가

이 과제에서는 아래 4개 상태로 나누면 작업하기 편합니다.

| 상태 | 이름 | 의미 |
|---|---|---|
| `00` | IDLE | 대기 상태 |
| `01` | LOAD | 데이터를 저장하는 상태 |
| `10` | REQ_TOGGLE | `req`를 반전시키는 상태 |
| `11` | WAIT_ACK | `ack` 응답을 기다리는 상태 |

---

## 6. 각 상태에서 해야 할 일

### IDLE

해야 할 일:

- 아무것도 성급하게 하지 말고 기다립니다.
- `ready = 1`이면 다음 상태로 넘어갑니다.

이동 조건:

- `ready = 0`이면 계속 IDLE
- `ready = 1`이면 LOAD

### LOAD

해야 할 일:

- `Data_to_send` 값을 내부 저장소(`data_reg`)에 넣습니다.

이동 조건:

- 데이터 저장 후 `REQ_TOGGLE`로 이동

### REQ_TOGGLE

해야 할 일:

- `req <= not req`로 값을 뒤집습니다.

이동 조건:

- `WAIT_ACK`로 이동

### WAIT_ACK

해야 할 일:

- `ack`와 `req`가 같아질 때까지 기다립니다.

이동 조건:

- `ack /= req`이면 계속 WAIT_ACK
- `ack = req`이면 IDLE로 복귀

## 7. 회로를 두 부분으로 나누면 쉬워집니다

문제를 한 번에 다 구현하려고 하면 헷갈리기 쉽습니다.
아래처럼 두 부분으로 나누면 이해와 구현이 쉬워집니다.

### 1. Control Unit

역할:

- 현재 상태를 기억
- 다음 상태를 결정
- `req`를 언제 바꿀지 결정
- 데이터 저장 타이밍(`load`) 결정

### 2. Datapath

역할:

- 데이터를 실제로 저장
- 저장한 값을 출력으로 유지

간단히 보면:

- Control Unit은 "언제 무엇을 할지" 결정
- Datapath는 "실제 데이터"를 다룸

## 8. 구현 순서

비개발자라도 아래 순서대로 하면 작업 흐름을 잡기 쉽습니다.

### 1단계. 입출력부터 정리

먼저 아래 신호가 필요한지 확인합니다.

- 입력: `Data_to_send`, `ready`, `ack`, `clk`, `reset_n`
- 출력: `Data`, `req`

### 2단계. 상태 이름 정하기

다음 4개 상태를 선언합니다.

- IDLE
- LOAD
- REQ_TOGGLE
- WAIT_ACK

### 3단계. 상태 전이표 작성

코드를 쓰기 전에 아래 내용을 표나 메모로 적습니다.

- 어떤 상태에서
- 어떤 조건일 때
- 어디로 이동하는지

이 단계가 정리되면 코드를 쓰기 훨씬 쉬워집니다.

### 4단계. 데이터 저장부 만들기

데이터를 바로 출력하지 말고, 내부 레지스터에 한 번 저장한 뒤 출력하도록 만듭니다.

이유:

- 값이 안정적으로 유지됩니다.
- 타이밍 문제를 줄일 수 있습니다.

### 5단계. `req` 제어 만들기

주의할 점:

- `req`는 필요할 때만 바꿔야 합니다.
- 보통 `REQ_TOGGLE` 상태에서만 바꾸는 것이 가장 안전합니다.

### 6단계. `ack` 확인 만들기

`WAIT_ACK` 상태에서:

- `ack = req`인지 확인합니다.
- 같아지면 전송 완료로 보고 IDLE로 돌아갑니다.

### 7단계. 리셋 동작 확인

리셋 시에는 보통 아래처럼 시작하게 만듭니다.

- `req = '0'`
- 상태 = IDLE

## 9. 구현하면서 자주 하는 실수

### 실수 1. 상태를 한 곳에서 여러 번 바꾸기

문제:

- 상태가 어디서 어떻게 바뀌는지 헷갈립니다.

권장:

- 상태 저장과 다음 상태 계산을 분리해서 생각합니다.

### 실수 2. `req`를 아무 때나 바꾸기

문제:

- 의도하지 않은 동작이나 타이밍 문제가 생길 수 있습니다.

권장:

- `REQ_TOGGLE` 상태에서만 `req`를 바꾸도록 설계합니다.

### 실수 3. 데이터를 바로 출력에 연결하기

문제:

- 출력 값이 흔들리거나 예상과 다르게 보일 수 있습니다.

권장:

- 내부 레지스터에 저장한 뒤 `Data`로 내보냅니다.

### 실수 4. 완료 조건을 잘못 쓰기

문제:

- `ack` 확인 기준이 틀리면 전송이 끝나지 않거나 너무 빨리 끝날 수 있습니다.

권장:

- `ack = req`일 때 handshake 완료로 판단합니다.

## 10. 테스트는 어떻게 하면 되는가

테스트의 목적은 "정말 의도한 순서대로 동작하는가"를 확인하는 것입니다.

아래 순서로 확인하면 됩니다.

### 기본 테스트

1. `ready = 0`일 때 아무 동작이 없는지 본다.
2. `ready = 1`이 되면 데이터 저장이 시작되는지 본다.
3. `req`가 이전 값에서 반전되는지 본다.
4. `ack`가 아직 다르면 기다리는지 본다.
5. `ack = req`가 되면 IDLE로 돌아가는지 본다.

### 추가 테스트

1. 여러 데이터를 연속으로 보내도 동작하는지 확인
2. `ack`가 늦게 와도 멈추지 않고 기다리는지 확인
3. 리셋 후 처음 상태로 정확히 돌아가는지 확인

## 11. 시뮬레이션에서 꼭 봐야 할 것

RTL 시뮬레이션에서 아래 항목을 확인합니다.

- 상태가 IDLE -> LOAD -> REQ_TOGGLE -> WAIT_ACK 순서로 가는지
- `req`가 토글되는 순간이 맞는지
- `ack`가 따라왔을 때 전송이 완료되는지
- `data_reg`가 올바른 값을 저장하는지
- 출력 `Data`가 안정적으로 유지되는지
